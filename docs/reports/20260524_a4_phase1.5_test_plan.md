## TL;DR

Phase 1 left A4 (hoist notation legend into the rules preamble) as the **strongest tier-score candidate** of the five audit-derived arms: mean Δ = +0.27 vs C0, best mean tier overall (3.93), directionally consistent with the audit's prediction (+0.0 to +0.2). It did NOT pass the pre-registered G1 gate because the 95% CI was [−0.53, +1.13] — too wide at N=15 (5 states × 3 runs) to rule out a null or negative true effect.

This plan re-runs **A4 vs C0 only**, at N=20 states × M=3 runs = **120 Haiku subagent calls total**, with the bench rebalanced to give us actual statistical power and to disambiguate the one place A4 lost in Phase 1 (oscillation).

Pre-committed decision: if A4 passes a tightened G1 gate (Δ ≥ +0.3 with CI excluding 0), it graduates to Phase 2 Gemma testing alongside A2. If it fails but doesn't actively regress, we defer the Gemma test and revisit when the production-data signal from `de7dc06` post-cutover sessions has more N. If it actively regresses, the plan ends and we go back to the audit.

## Why A4, why now

### What A4 is

The notation legend currently lives **inline with the board data**:

```
CURRENT GAME:
NOTATION: rank+suit (A 2-9 T J Q K; H D C S). ?? = face-down. In each column the top of the stack is the rightmost card.

FOUNDATIONS:   H: --   D: --   C: --   S: --
...
```

A4 hoists this legend into the **rules preamble**, between the KLONDIKE SOLITAIRE RULES block and the STRATEGY GUIDANCE block, so the model sees it as part of the standing instructions rather than as a header on the data payload. The CURRENT GAME section in A4 starts directly with FOUNDATIONS, no notation line.

Rationale from the original audit: notation rules are reference material the model uses while parsing data. Putting them adjacent to the data forces the model to re-parse the legend each turn rather than treating it as a fixed grammar. Hoisting it to the preamble matches the standard prompt-engineering principle of "rules above data, data above task."

### Why it's our best remaining candidate

Phase 1 outcomes summarized:

| Arm | Mean Δ vs C0 | 95% CI | Output reduction | Verdict |
|---|---:|---|---:|---|
| A1 (drop reasoningTrail) | −0.53 | [−1.67, +0.53] | 0.07× (small) | Directionally negative; harvester shipped it anyway |
| A2 (drop prose CoT) | −0.47 | [−1.53, +0.60] | **19×** | Mixed: huge cost win, possible quality loss |
| A3 (A1 + A2) | −0.13 | [−0.73, +0.27] | **19×** | Bisects A1 and A2 |
| **A4 (hoist notation)** | **+0.27** | [−0.53, +1.13] | 1.1× (none) | **Best point estimate; matches audit prediction** |

A4 is the only arm where the data and the audit agreed *directionally*. It's also the lowest-risk arm to graduate because it does not change the response schema (so JSON validity won't regress) and does not remove any content the model uses for reasoning — it only moves a reference table closer to the rules.

### Why Haiku, not Gemma

Same reasons as Phase 1: Gemini quota is expensive and slow; Haiku 4.5 (via Claude Code subagent) is free and confirms the *direction* of the effect cheaply. The Phase 1 result that surprised us most — A1's negative effect on Haiku — would have been a wasted Gemma run if we'd skipped the proxy step. A4 deserves the same triage rigor.

## Hypotheses (pre-registered)

### H1 (primary)

**A4 produces a higher mean tier-score than C0, with 95% paired-bootstrap CI excluding 0.**

Specifically: mean Δ ≥ +0.3 tier with CI lower bound > 0.

The +0.3 threshold is below the +0.4 we used in Phase 1 because (a) we now have a real point estimate (+0.27) to anchor against, not a pre-audit guess, and (b) the cost of a false-negative on A4 is high (we hold the only quality-positive arm). It's still above 0 so we don't ship noise.

### H2 (directional safety)

**A4 does not underperform C0 on oscillation states.**

Phase 1 had A4 at 3.33 vs C0 at 3.83 on oscillation — a 0.5 tier deficit, but on only 2 oscillation states. With 5 oscillation states in this run, mean A4_osc should be ≥ C0_osc − 0.3.

If H2 fails (A4 is meaningfully worse on oscillation even at N=5), we have a real concern: hoisting notation might cost the model context-switching cycles when the board is in a low-progress repetitive state. We'd want to understand which states regress before any Gemma test.

### H3 (replication)

**A4 maintains its endgame advantage at higher N.**

Phase 1 had A4 hitting 6.00 (perfect foundation choice) on the one endgame state. With 5 endgame states, A4_endgame should be ≥ 5.0. If A4 drops below 5.0 on endgame, the Phase 1 endgame result was probably a single-state artifact.

### H4 (output-size sanity)

**A4 output size is approximately C0 output size (within 5%).**

A4 doesn't change the response schema, so output length should be unchanged. If A4 output is meaningfully different (>5%), something else changed — re-examine the renderer.

### What we do NOT predict

- **Per-state direction**: A4 will lose on some individual states. That's fine; the test is on means.
- **JSON validity**: both arms should be 100% (Haiku handles both layouts cleanly per Phase 1). If A4 drops below 100%, surface it but don't make it a gate.
- **Reversal-pick rate**: A4 was 7% in Phase 1. We expect ~7% again. Not a gate.

## Experimental design

### Arms

Just **two**: C0 (control, current production format including the new `de7dc06` changes) and A4 (notation hoisted to preamble).

C0 baseline is the **current production prompt format** as observed in `data/raw/solitaire-ai-log-359e1f-1779581014927.json` and its sibling — i.e., the `0462323c…` template hash. This is **different from Phase 1 C0** (which used the pre-`de7dc06` format). We rebuild the C0 renderer to match the new template, otherwise we're comparing A4 to a stale baseline.

This means A4 renderer also needs updating: hoist notation from the new C0, not the old one.

### Bench composition (N=20 states)

Rebalanced from Phase 1's 6/6/4/4 split:

| Category | Count | Rationale |
|---:|---:|---|
| Early (≥18 face-down) | 4 | Phase 1 had A4 ahead here (+1.0 tier); maintain |
| Midgame (10–17 face-down) | 7 | Largest production share; was tied in Phase 1 (5.00 = forced reveal) |
| Endgame (≤9 face-down) | 4 | H3 test ground; need ≥4 to see if 6.00 generalizes |
| **Oscillation** | **5** | H2 test ground; we lost here on 1 of 2 in Phase 1, need ≥5 to know if real |

Of the 5 oscillation states, **at least 3 must have a `tableau_to_foundation` move available** (the smoking-gun pattern from `oscillation-bffb5d9cc178`). The remaining 2 can be pure shuffle-loop states.

Bench builder: extend `/Users/chayut/repos/solitaire-analytics/experiments/prompt_audit_2026_05_24/build_bench.py` with a new `--n 20 --bias oscillation-with-foundation` flag, or write a new builder if the existing one is too crufty.

### Run plan

- **120 total calls**: 20 states × 3 runs × 2 arms
- Subagent simulation, same harness as Phase 1 (one fresh subagent per call, write response to disk)
- **Batching**: 15 subagents per orchestrator turn, ~8 turns total. ~3-4 hours wall-time at Phase 1 cadence.
- **Reuse Phase 1 infrastructure**: `render_to_disk.py`, `score_raw.py`, `aggregate.py` should work with minimal change — just point them at the new bench and the new arm set.

### Token budget

~120 calls × ~26K tokens/call = **~3.1M tokens** of subagent simulation. Fits comfortably in a session.

## Pre-registered decision rules

These are committed **before** seeing the results. No retconning.

### If H1 passes (Δ ≥ +0.3, CI excludes 0)

**Graduate A4 to Phase 2 Gemma testing.** Pair it with A2 in the Gemma run. Phase 2 design uses Gemini quota and tests A4 + A2 vs C0 head-to-head against the new `de7dc06` baseline.

### If H1 fails but H2 holds (no oscillation regression)

**Defer A4.** The Haiku data is consistent with A4 being mildly positive but not strongly so. Wait for `de7dc06` production data to accumulate (target: ≥10 post-cutover sessions, so we have an apples-to-apples comparison with pre-cutover gameplay). Re-evaluate then.

This is the most likely outcome and that's OK — A4 is a low-risk change, and the observational signal from production traffic may be cheaper than running Gemma sweeps.

### If H2 fails (A4 regresses on oscillation)

**Stop and investigate.** Look at the specific oscillation states A4 lost on. Hypothesize why (e.g., does A4 cause the model to pick more shuffles when faced with no progress?). If we can characterize the failure mode, we know what to fix. If we can't, A4 is dead.

### If H3 fails (no endgame advantage)

Note in the report but do not change the H1 decision. The endgame result was a Phase 1 surprise; if it doesn't replicate, we just don't claim it anymore.

### If H4 fails (output size diverges)

Treat as a bug in the renderer. Fix and re-run. Do not interpret as a real A4 effect.

## What could go wrong (and what to do)

| Risk | Likelihood | Mitigation / interpretation |
|---|---|---|
| Phase 1 A4 result was a fluke; A4 actually has zero true effect | Moderate | At N=20 states, CI tightens by ~√(20/5) = 2× → ±0.5 instead of ±1.0. If true effect is +0.27, this run has ~50% power to detect it. Be honest if CI still crosses 0. |
| Haiku result doesn't transfer to Gemma | Real, unmitigated here | This is the known limitation; both Phase 1 and Phase 1.5 are proxies. Only Gemma data settles it. That's why "graduate" doesn't mean "ship", it means "Phase 2 Gemma test next". |
| New C0 baseline (`0462323c…` template) behaves very differently from old C0 | Moderate | The new C0 has the A1 change already shipped. So we're testing A4 *on top of* A1. That's actually the realistic deployment scenario — but make sure the report notes this. |
| Subagent simulation introduces systematic bias one of the arms benefits from | Low | Subagent system prompt is identical across arms; only the prompt-under-test differs. Phase 1 didn't show obvious arm-side bias. |
| Oscillation bias in the bench picks states A4 was always going to lose on | Low-moderate | Bench is built from production data, not curated. If A4 loses on 4 of 5 oscillation states the loss is real, not a setup. |
| The "oscillation-with-foundation" pattern is too rare in production data | Low | We have 8000+ interactions; should be plenty. If the bench builder can't find 3, document it and adjust threshold. |

## Relationship to the production-data analysis

This Phase 1.5 plan is the **experimental** side. The **observational** side runs in parallel: as more `de7dc06` (A1-shipped) sessions land in `data/raw/`, we'll get to compare pre-cutover vs post-cutover production gameplay directly.

The two should reach a consistent verdict on A1's effect (Haiku Phase 1 says negative; production data should confirm or refute). If they disagree, the production data wins — it's the real-world signal.

For A4 the analogy doesn't apply yet — A4 is not in production. The Haiku Phase 1.5 result is the only signal we'll have until either (a) the harvester team agrees to ship A4 based on Phase 1.5 + Phase 2 evidence, or (b) we run Phase 2 Gemma ourselves to triangulate.

## Approach notes / lessons applied from Phase 1

- **Two-arm design, not five**: Phase 1 spent 60% of calls on arms (A1, A3) that turned out to be uninformative or negative. Two-arm cuts the budget and concentrates power on the question we care about.
- **Bigger N per category, not bigger M per state**: Phase 1's between-state variance was much larger than within-state variance. Adding more states buys more power than adding more runs per state.
- **Pre-register specific numbers**: Phase 1 had "Δ ≥ +0.4, CI excludes 0" pre-registered and we lived by it. This plan does the same. The +0.3 threshold here is justified above (anchored to the Phase 1 point estimate).
- **Don't re-test settled questions**: JSON validity, reversal-pick rate, output-size on a no-schema-change arm — all reportable, none gating.
- **Stage the costly thing last**: Gemma is the expensive step. Don't burn it confirming what a cheap Haiku run can confirm first.
